2. Architecture Overview

The system is cleanly decoupled into two main components: a Static Frontend and a Serverless API.

The Frontend (Client-Side)

Hosted entirely on GitHub Pages, making it globally cached and extremely fast. It consists of two interfaces:

  • Customer Form (index.html): Optimized for mobile browsers (viewport locked). Uses JS Geolocation API to fetch GPS coords.
  • Admin Dashboard (admin.html): Secured via local browser session storage. Renders data dynamically into tables and charts using pure JS DOM manipulation.

The Backend (Server-Side)

Google Apps Script is deployed as a "Web App", acting as a REST API. When doPost() or doGet() triggers, it parses the JSON payload, routes it to the specific function (e.g. addFeedback()), and performs read/write operations on the Google Sheet using the SpreadsheetApp service. Finally, it returns a ContentService.MimeType.JSON response back to the client.

3. Technical Data Flow (Step-by-Step)

1
QR Code Scan & Form Initialization

Customer scans a branch-specific QR code. The URL contains a URL Parameter (e.g., ?branch=025). JavaScript intercepts this parameter to lock the feedback to the specific branch. JS concurrently requests GPS permissions via navigator.geolocation.

2
Data Submission & CORS Handling

Upon clicking submit, JS packages the rating, tags, comments, and coordinates into a JSON string. It uses the fetch() API to send an asynchronous POST request. The Google Script natively handles CORS requests, allowing GitHub Pages to securely write data cross-origin.

3
Database Write (Google Sheets)

The backend intercepts the request in the doPost(e) event object. It parses the body, connects to the "Feedback" sheet tab, and executes sheet.appendRow(), pushing a new timestamped row. A success JSON response is sent back to clear the customer's UI.

4
Automated Reporting (Cron Jobs)

The backend utilizes Google's Time-Driven Triggers (ScriptApp). The setup() function initializes a cron job that runs daily between 8:00 AM - 9:00 AM. This background worker checks the "Settings" sheet tab for the Admin's configured frequency (Daily, Weekly, Monthly) and uses MailApp.sendEmail() to dispatch a dynamically compiled HTML email report containing calculated analytics and a CSV attachment Blob.

4. Security & Optimization Concepts Used

Data Type Sanitization: A critical safeguard implemented in admin.js ensures that branch codes parsed from Google Sheets (which aggressively auto-casts values to numbers) are strictly cast back using String(value) before performing Regex or UI replacements, preventing application crashes.
  • Debouncing / Event Delegation: UI events (like search filters) are heavily optimized to prevent excessive DOM reflows.
  • Lightweight State Management: Instead of Redux, the app uses a centralized state object (let state = { branches: [], page: 1, ... }) to manage the source of truth for the dashboard.
  • Local Storage Auth: The admin portal uses sessionStorage and localStorage to maintain admin state without requiring complex JWT token databases.
