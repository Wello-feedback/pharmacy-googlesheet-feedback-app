2. Architecture Overview (System kaam kaise karta hai?)

Is system ke 2 main hisse (parts) hain: Frontend (Jo user ko dikhta hai) aur Backend (Jo piche background me data process karta hai).

A) The Frontend (Client-Side)

Ye wo code hai jo GitHub Pages par live hai. Isme 2 main screens hain:

  • Customer Form (index.html): Ye screen specially mobile ke liye design ki gayi hai. Jab customer QR scan karta hai, toh URL se us branch ka Code nikal liya jata hai. Uske baad browser ka GPS feature use karke location background me fetch hoti hai.
  • Admin Dashboard (admin.html): Ye admin panel hai jise secure karne ke liye sessionStorage aur localStorage ka use kiya gaya hai. Isme JavaScript direct backend se data mangwati hai aur bina page reload kiye Table aur Charts generate karti hai (Dynamic DOM manipulation).

B) The Backend (Google Apps Script API)

Aapne jo code Apps Script me daala hai, wo ek REST API ki tarah kaam karta hai. Jab bhi Frontend se koi Request (GET ya POST) aati hai, toh handleRequest() function usko padhta hai aur check karta hai ki "action" kya hai (jaise: addFeedback, getBranches, etc.). Phir wo SpreadsheetApp use karke Google Sheet me se row delete, add, ya read karta hai aur JSON format me result wapas bhej deta hai.

3. Technical Data Flow (Step-by-Step Technical Process)

1
QR Scan & Initialization (Customer Entry)

Customer ek QR code scan karta hai. URL ke aakhir me ?branch=025 likha hota hai. Javascript is branch parameter ko padh kar branch ka naam auto-fill kar deti hai. Usi time, navigator.geolocation API customer ki current GPS location nikalne ki koshish karti hai.

2
Data Submit (API Calling)

Jab customer "Submit" dabata hai, Javascript unke rating, tags, comment aur location ka ek JSON object banati hai. Phir Javascript fetch() API ka use karke is JSON data ko Google Apps Script ke Web App URL par POST request ke roop me bhejti hai. (Isme CORS handle karne ke liye text/plain content-type use hota hai).

3
Database Writing (Backend me Data Save hona)

Google Apps Script is POST request ko doPost(e) event me catch karti hai. JSON ko parse karti hai aur usme current Date & Time (Timestamp) jod kar Google Sheet (Feedback sheet) me ek nayi row append kar deti hai: sheet.appendRow([date, name, mobile...]). Phir frontend ko {success: true} bhej deti hai jisse "Thank you" screen aa jati hai.

4
Automated Email Reporting (Cron Jobs / Triggers)

Backend me ek Timer chalta hai (Google Time-driven trigger). Ye trigger roj subah 8 baje automatedReportTask() function ko jagata hai. Ye function "Settings" tab me check karta hai ki mail ki frequency kya hai (Daily, Weekly ya Monthly). Agar aaj mail bhejne ka din hai, toh wo Data collect karke ek HTML Email template banata hai, ek CSV file generate karta hai (Blob form me), aur MailApp.sendEmail() ke zariye admin ko automatically bhej deta hai.

4. Security & Error Handling (Bugs se bachne ki techniques)

Data Casting Bug Fix (Jo number wali problem aayi thi):
Google Sheets kabhi-kabhi branch names ko (jaise '102') Text ki jagah Number samajh leta hai. Jab Javascript me number par `.replace()` function chalaya jata hai, toh system crash (loading loop) ho jata hai. Isko theek karne ke liye code me humne Strict String Casting String(value) ka use kiya hai taaki application crash na ho.
  • Asynchronous Await (async/await): API se data aane me time lagta hai, isliye humne promises ka use kiya hai (await fetch()) taaki data aane tak system ruka rahe aur galat data na dikhaye.
  • Local Storage Security: Admin login details localStorage aur sessionStorage me rakhi gayi hain. Agar aap tab close karte hain, toh session expire ho jata hai aur wapas aane par aapse dobara password manga jata hai. Ye simple but strong security hai.
  • Debouncing on Search: Agar aap Table me kisi customer ka naam search karte hain, toh table ekdam lag nahi karta, kyunki frontend ka architecture sirf wohi data render karta hai jo match hota hai.
